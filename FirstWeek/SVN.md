# SVN

## 标准目录布局
>- trunk（主干）
>- branches（分支）
>   - 维护分支
>   - 新特性分支
>   - 缺陷修复分支
>   - ...
>- tags（标签/快照）
>   - 版本发布留档

## 开发方式

### 集中式
>- 做法
>   - 使用trunk进行主要开发
>   - 某一阶段结束后，放入tags存档
>   - 已发行版本出现问题时，找到对应tags，发布branches进行修改
>- 优点
>   - 管理简单
>- 缺点
>   - 模块较多且小团队比较多时，很容易产生代码冲突

### 分散式
>- 做法
>   - 使用branches进行主要开发
>   - 某一阶段结束后，放入tags存档，并在branches中创建新阶段的目录
>   - 选择性的进行代码合并，放入trunk中进行发布
>- 优点
>   - 各自开发独立，互不影响
>- 缺点
>   - 管理复杂，代码合并麻烦

## 友田方式

### 目录
>- trunk：用于版本开发提交代码
>- branches：用于定制版本和Bug修复
>- tags：用于记录和保存已发布版本

### 开发过程
>- 初期集中开发
>- 后期分散开发
